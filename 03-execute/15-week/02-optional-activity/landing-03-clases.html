Diagrama de clases para schedule-sena

El diagrama de clases representa la estructura estática del dominio: clases, atributos, operaciones, relaciones y multiplicidades. En este proyecto se parte de las entidades reales identificadas, sin inventar estructuras externas ni copiar tablas de base de datos automáticamente.

Enfoque

Primero modelo conceptual desde el dominio, luego relaciones, después atributos y operaciones. No se empieza copiando código ni tablas completas del proyecto.

Clases candidatas del dominio

Desde el backend inspeccionado y el contexto SENA se identifican estas clases candidatas para iniciar el modelado. El aprendiz debe justificar la inclusión o exclusión de cada una:

Instructor Ficha AmbienteFormacion Horario Observacion ProgramaFormacion Disponibilidad

Solo se incluye una clase si tiene responsabilidad propia identificable dentro del dominio del sistema. No agregar clases por convención o para completar un número.

Buenas técnicas de modelado

  • Partir de conceptos del dominio, no de pantallas ni tablas de base de datos.
  • Nombrar clases en singular: Instructor, no Instructores.
  • Evitar clases genéricas sin responsabilidad clara.
  • No inventar atributos que no estén confirmados en el proyecto.
  • Diferenciar asociación de composición con base en el ciclo de vida de las partes.
  • Validar multiplicidades contra las reglas de negocio identificadas.
  • No saturar el diagrama con getters, setters ni detalles de implementación.
  • Mantener trazabilidad: cada clase debe poder rastrearse a un caso de uso o regla del dominio.

Técnica incremental para construir el diagrama

1Extraer sustantivos
Ambiente, instructor, ficha, horario, observación del dominio.
2Validar dominio
Quitar conceptos de pantalla, tecnología o infraestructura.
3Agregar atributos
Solo los relevantes para comprender el modelo del negocio.
4Definir relaciones
Asociación, composición, agregación o herencia con justificación.
5Validar reglas
Conflictos, estados, inmutabilidad y multiplicidades correctas.

Ejemplo parcial — no es la solución

Este ejemplo es intencionalmente incompleto. El aprendiz debe agregar Observacion, ProgramaFormacion y Disponibilidad con multiplicidades y operaciones justificadas desde su análisis propio.

Relaciones probables a validar

Relación Interpretación Tipo UML sugerido

Horario → InstructorUn horario tiene asignado un instructor.Asociación
Horario → FichaUn horario corresponde a una ficha específica.Asociación
Horario → ObservacionUn horario puede tener observaciones registradas.A justificar: asociación o composición
ProgramaFormacion → FichaUn programa tiene varias fichas activas.Asociación o agregación
Instructor → DisponibilidadUn instructor tiene disponibilidad horaria registrada.Asociación o composición

No asumir composición sin justificar el ciclo de vida. En el backend, Observacion referencia a Horario y es inmutable una vez registrada.

Criterios de entrega

  • Archivo .drawio con el diagrama de clases completo.
  • Exportación del diagrama en formato PDF o SVG.
  • Documento Word con la plantilla de justificación diligenciada para cada clase incluida.
  • Las clases deben corresponder al dominio real del proyecto schedule-sena.
  • Las relaciones y multiplicidades deben estar justificadas con base en reglas del negocio.
  • La documentación debe ser redacción propia, coherente con los casos de uso de la Landing 2.
Advertencia sobre uso de inteligencia artificial: la IA puede apoyar la comprensión de conceptos UML y resolver dudas puntuales. No puede construir el diagrama ni diligenciar la justificación de clases en lugar del aprendiz. Si se detecta que la evidencia fue generada completamente por IA sin trabajo manual verificable, el aprendiz deberá realizar una actividad adicional de sustentación y ajuste manual.
Reto final: revisa el diagrama de casos de uso que construiste en el taller anterior. ¿Cuántas clases adicionales podrías justificar a partir de esos casos de uso?
