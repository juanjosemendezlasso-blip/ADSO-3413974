El contrato técnico entre

de desarrollo
¿Qué
es?
Un SRS (Software Requirements Specification) es el documento formal que describe qué debe hacer un sistema de software, bajo qué condiciones debe operar y qué restricciones debe cumplir — sin prescribir cómo implementarlo. Es la base del contrato entre stakeholders y desarrolladores.
Objetivos
01
01
Comunicar
Establece un lenguaje común entre clientes, usuarios y desarrolladores, reduciendo malentendidos.
02
Validar
Permite verificar que el sistema construido cumple lo acordado antes de entrar a producción.
03
Controlar
Sirve de referencia para gestión del cambio, estimaciones de costo y planificación del proyecto.
Estructura IEEE 830
02
Secciones del documento SRS ESTÁNDAR IEEE 830 / ISO/IEC 29148
1
Introducción

Propósito, alcance, definiciones, acrónimos, referencias y visión general del documento.

2
Descripción General

Perspectiva del producto, funciones principales, características del usuario, restricciones generales y supuestos.

3
Requisitos Específicos

Requisitos funcionales, no funcionales, de interfaz, de rendimiento y restricciones de diseño. Es la sección más extensa.

4
Apéndices y Anexos

Modelos adicionales, glosario extendido, diagramas, prototipos y cualquier información de soporte.

Tipos de Requisitos
03
Funcionales
¿Qué hace el sistema? Comportamientos, funciones, casos de uso. Ej: "El sistema debe permitir registrar usuarios con correo y contraseña."
No Funcionales
¿Cómo lo hace? Rendimiento, seguridad, disponibilidad, escalabilidad. Ej: "El sistema debe responder en menos de 2 segundos para el 95% de las solicitudes."
Interfaz
¿Con qué interactúa? Interfaces de usuario, hardware, software externo, comunicaciones. Ej: "El sistema debe integrarse con la API REST del ERP corporativo."
Restricciones
¿Bajo qué límites? Tecnológicos, legales, presupuestarios, normativos. Ej: "El sistema debe cumplir con la Ley 1581 de protección de datos de Colombia."
Cualidades de un buen SRS
04
◎ Correcto Cada requisito refleja fielmente la necesidad real del usuario o cliente.
◈ Inequívoco Una sola interpretación posible. Sin ambigüedades ni términos vagos.
◉ Completo Cubre todas las funciones, condiciones de entrada, salida y restricciones del sistema.
▣ Verificable Cada requisito puede ser probado o validado con métodos definidos y medibles.
Proceso de Elaboración
05
1
Elicitación
Entrevistas, talleres, observación, prototipos para capturar necesidades.
2
Análisis
Identificar conflictos, priorizar y clasificar los requisitos obtenidos.
3
Especificación
Redactar el SRS con lenguaje formal, preciso y estructurado.
4
Validación
Revisión con stakeholders para confirmar que el documento es correcto.
5
Gestión
Control de cambios, trazabilidad y versionado durante todo el ciclo.
Errores vs. Buenas Prácticas
06
❌ Errores Comunes
Requisitos vagos o ambiguos
Mezclar soluciones con necesidades
Ignorar stakeholders clave
No versionar el documento
Requisitos no verificables
Omitir restricciones del negocio
✔ Buenas Prácticas
Usar plantilla IEEE 830 o ISO 29148
Priorizar con MoSCoW o Kano
Involucrar usuarios reales
Mantener trazabilidad completa
Revisar con todos los stakeholders
Actualizar ante cada cambio aprobado
